# Escrevendo histórias de usuário

### User Stories

- Abordagem para estuturar requerimentos a partir da visão do usuário
- Fórmula para capturar **User Stories**:
  - Saber quem é o usuário que deseja funcionalidade
  - Saber o que o usuário deseja
  - Entender a motivação e razão
  - Princípio 3C (Cartão, conversa, confirmação)
 
  <img width="371" height="186" alt="image" src="https://github.com/user-attachments/assets/0ab52df1-fec1-4e6f-9616-f6435bcf2233" />

- Uso de outros artefatos é bem vindo desde que contribua para compreensão: Diagramas, fluxogramas, mindmaps, protótipos, etc...
- A discussão com a equipe é essencial para refinamento!

---

### Características de uma User Story

- INVEST
  - Independente
  - Negociável
  - Valiosa
  - Estimável
  - Pequena (Small)
  - Testável
 
  ---

  ### Critérios de aceite

  - Confirmação se requisitos foram atendidos
    - Em forme de sentenças como um checklist
    - Uso de linguagem Guerkin (Given/Then/When)

  <img width="397" height="288" alt="image" src="https://github.com/user-attachments/assets/411168b8-4585-461b-baf6-05b230b8683c" />

### No que devemos prestar atenção?

- Sempre analise se a história setá focado no usuário ou cliente
- Histórias de usuários != requisitos bem definidos e documentados
- Pense em INVEST
- Pense nos requisitos de qualidade que não são pensadas pelo cliente
- A User Storie pode e deve ser discutida pela equipe e criada colaborativamente
- Comunicação é essencial!

---

### Um exemplo de US

1. Entrega de valor: objetivo da funcionalidade
2. Narrativa do usuário
3. Requisitos técnicos
   A. Atores
   B. Interfaces/fluxos
   C. Dados
   D. Regras e negócios
   E. Ambiente
   F. Critérios de aceite
